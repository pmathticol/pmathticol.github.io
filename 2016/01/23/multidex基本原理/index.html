multidex基本原理

multidex基本原理

multidex

1.出现原因

下面所说的的都是在android 5.0以下
multidex主要是因为两个情况：
- 第一个情况是一个dex的情况下方法数过多，如果一个dex超过了65536个方法，就会安装不成功，这个是本事dex格式限制，应用层无法更改，一个dex最多只能容纳65536个方法。
- 第二个情况是在apk安装的时候会有一个dexopt的过程，这个是dexopt过程本身分配的内存的限制，5.0以下分别经历了4M/5M/8M/16M的限制，但是不能从根本上解决问题，假设方法没有达到上限，但是dexopt的时候超过这个内存限制，也会触发无法安装的错误。从这个错误可以看到，假设如果我们分成的两个包

上面两种情况，第一种情况，采用multidex的方案后可以解决，但是第二个问题采用multidex方案后，依然还是可能有bug的，具体看issure 22586和 Isseure 78035

2.具体过程

取自这篇文章
用一句话来说，multidex主要是用来讲各个分dex信息添加加到DexClassLoader中的DexPathList的Elements数组当中，这个数组的格式包含了file，ZipFile，DexFile格式的所有信息。然后通过findclass就能够找到所有的类了，应用层就能运行了。
主要包含两部分，第一部分是从apk包中找寻各个分dex(class2.dex,class3.dex....classN.dex),然后解压出来，在压缩至zip格式的文件放在file列表里面，之前做的一个优化就是这个步骤,图中installSecondary上面的部分(省解压和压缩步骤)，得到这个zip格式列表之后，在installSecondaryDexes(loader，dexDir，files)函数里面装载这个files列表，files列表就是存放的dex，zip格式存在，dexDir是一个app的得到存放dex的一个私有的缓存目录，loader是BaseClassLoader的子类，也就是DexClassLoader。安装的过程其实就是将把包含dex信息转换成DexPathList的dexElements数组，然后添加到DexClassLoader中当前DexPathList中的已知的dexElements数组后面即可。

所以如果不用multidex方案，直接自己用DexClassLoader将分包好的分dex加到这个dexElements数组后面也是可以的。

应用层安装核心的函数(v19版来说)：

  1. private static void install(ClassLoader loader, List<File> additionalClassPathEntries,
  2.                 File optimizedDirectory)
  3.                         throws IllegalArgumentException, IllegalAccessException,
  4.                         NoSuchFieldException, InvocationTargetException, NoSuchMethodException {
  5.             /* The patched class loader is expected to be a descendant of
  6.              * dalvik.system.BaseDexClassLoader. We modify its
  7.              * dalvik.system.DexPathList pathList field to append additional DEX
  8.              * file entries.
  9.              */
  10.             Field pathListField = findField(loader, "pathList");//从DexClassLoader找到DexPathlist pathlist这个字段
  11.             Object dexPathList = pathListField.get(loader);
  12.             ArrayList<IOException> suppressedExceptions = new ArrayList<IOException>();
  14.             //把makeDexElements返回后的分dex信息的dexElements[]数组，这个是属于DexPathList类的字段，数组里面每一个元素封装了zipfile，dexfile等信息，
  15.             //expandFieldArray是将分dex信息的dexElements数组添加到当前dexElements[]这个数组后面，
  16.             expandFieldArray(dexPathList, "dexElements", makeDexElements(dexPathList,
  17.                     new ArrayList<File>(additionalClassPathEntries), optimizedDirectory,
  18.                     suppressedExceptions));
  20.             if (suppressedExceptions.size() > 0) {
  21.                 for (IOException e : suppressedExceptions) {
  22.                     Log.w(TAG, "Exception in makeDexElement", e);
  23.                 }
  24.                 Field suppressedExceptionsField =
  25.                         findField(loader, "dexElementsSuppressedExceptions");
  26.                 IOException[] dexElementsSuppressedExceptions =
  27.                         (IOException[]) suppressedExceptionsField.get(loader);
  29.                 if (dexElementsSuppressedExceptions == null) {
  30.                     dexElementsSuppressedExceptions =
  31.                             suppressedExceptions.toArray(
  32.                                     new IOException[suppressedExceptions.size()]);
  33.                 } else {
  34.                     IOException[] combined =
  35.                             new IOException[suppressedExceptions.size() +
  36.                                             dexElementsSuppressedExceptions.length];
  37.                     suppressedExceptions.toArray(combined);
  38.                     System.arraycopy(dexElementsSuppressedExceptions, 0, combined,
  39.                             suppressedExceptions.size(), dexElementsSuppressedExceptions.length);
  40.                     dexElementsSuppressedExceptions = combined;
  41.                 }
  43.                 suppressedExceptionsField.set(loader, dexElementsSuppressedExceptions);
  44.             }
  45.         }
